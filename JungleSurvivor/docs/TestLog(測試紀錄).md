## 關鍵發現與結論

### 發現 1：通用模型的植物辨識能力有限

**現象**：直接問模型「這是什麼植物？」，對常見但不起眼的植物（車前草）辨識失敗。

**結論**：

- 不能依賴模型「自己認識」植物
- 模型的真正強項是 **描述形態特徵**（葉形、葉脈、生長環境等描述都很準確）
- 應該讓模型做 **「觀察描述者」而不是「辨識專家」**

**開發指引**：

- 辨識任務交給「特徵描述 → 知識庫比對」的 pipeline
- 模型負責描述，知識庫負責匹配

### 發現 2：Few-shot Prompt 有致命的誘導偏差

**現象**：在 Prompt 裡列出 3 種參考植物後，模型會硬從中選一個，即使照片中的植物完全不在選項中。

**結論**：

- **絕對不能** 用「從以下 N 種植物中選一個」的封閉式 Prompt 做通用辨識
- Few-shot 只適合用在 **已經鎖定候選物種** 的第二層鑑別（例如「這是芋頭還是姑婆芋？」）
- 第三層的通用辨識必須用 **開放式 Prompt + 安全閾值**

**開發指引**：

- 第一層（危險快篩）：封閉式比對 → OK，因為問的是「像不像」而不是「是不是」
- 第二層（混淆鑑別）：封閉式二選一 → OK，但必須允許「無法確定」的選項
- 第三層（通用辨識）：**開放式** → 必須，不能給選項

### 發現 3：危險快篩比對模式有效

**現象**：餵入姑婆芋照片，用比對清單的方式，模型給出 65% 相似度並觸發警告。

**結論**：

- 比對式 Prompt（「像不像」）比辨識式 Prompt（「是什麼」）表現更好
- 65% 偏低（畢竟真的是姑婆芋），需要加強特徵描述來提升信心度
- 模型能正確排除不相關物種（菇類 0%、咬人貓 15%），選擇性比對是有效的

**開發指引**：

- 每種有毒物種的清單需要 **更詳細的形態特徵描述**（不能只寫一行）
- 加入環境上下文可以進一步提升準確度

### 發現 4：安全閾值和保守策略是必須的

**現象**：要求模型「信心度 < 70% 標不確定」後，模型確實在不確定時誠實回答。

**結論**：

- 「寧可說不知道，也不要猜錯」這條指令模型會遵守
- 安全閾值是整個系統安全性的最後防線
- 不同層級應有不同的閾值設定

**開發指引**：

- 第一層（危險快篩）：閾值 60%（寧可誤報也不漏報）
- 第二層（混淆鑑別）：閾值 80%（無法確定就當有毒）
- 第三層（通用辨識）：閾值 70%（不確定就建議不碰）

### 發現 5：中文回覆需要更多 token

**現象**：`max_new_tokens=512` 時中文回覆被截斷。

**結論**：

- 中文字比英文佔更多 token（約 1.5-2 倍）
- 設定 `max_new_tokens=2048` 可覆蓋大部分回覆
- 結構化的回覆（含表格、分項列點）特別吃 token

**開發指引**：

- 預設 `max_new_tokens=2048`
- 要求 JSON 輸出時可能可以減少（JSON 比自然語言簡潔）
- 危險快篩因為要比對多種物種，可能需要 `max_new_tokens=4096`

### 發現 6：多 GPU 環境需要注意 device 分配

**現象**：Kaggle T4 x2 環境中，`device_map="auto"` 導致張量分散在兩張 GPU 上，推理時報錯。

**結論**：

- E2B 模型夠小，一張 T4（16GB）足以容納
- 強制 `device_map={"": "cuda:0"}` 最簡單

**開發指引**：

- 開發階段固定 `device_map={"": "cuda:0"}`
- 部署到手機/邊緣設備時本來就只有一個設備，不會有此問題

## Prompt 改進方向

### 改進 1：有毒物種清單需要詳細特徵描述

**之前（太簡略）：**

```
1. 姑婆芋 (Alocasia odora) — 全株有毒，誤食致口腔灼傷、腹瀉
```

**之後（應該這樣寫）：**

```
1. 姑婆芋 (Alocasia odora)
   毒性：全株有毒，汁液含草酸鈣針晶，誤食致口腔灼傷、腹瀉、喉嚨腫脹
   葉形：大型心形或箭形，長 50-90cm，寬 40-60cm
   葉面：光滑發亮，深綠色，水珠不成珠狀（會攤平）
   葉脈：明顯的羽狀脈，主脈粗壯突出
   葉尖：朝上或水平指向
   葉柄：粗壯，長 50-100cm，綠色常帶紫暈，接在葉片邊緣
   莖部：粗短直立莖，多汁
   生長環境：林下、溪邊、潮濕陰暗處，海拔 0-1500m
   易混淆：芋頭 (Colocasia esculenta)
```

**預期效果**：模型有更多特徵可比對，信心度從 65% 提升到 80%+

### 改進 2：加入環境上下文前綴

在所有 Prompt 的最前面加入環境資訊：

```
【環境上下文】
地點：台灣
氣候帶：亞熱帶
海拔：約 {altitude}m
植被帶：{vegetation_zone}
季節：{season}
此環境中常見的大型植物主要為：{common_species_list}
```

**預期效果**：模型自動縮小候選範圍，不會往不可能的方向猜

### 改進 3：要求模型先觀察再判斷（Chain of Thought）

```
請按照以下步驟分析：
步驟 1：仔細觀察照片，描述你看到的形態特徵
步驟 2：將觀察到的特徵與清單中的每種植物逐一比對
步驟 3：給出比對結果和相似度
步驟 4：如果有任何特徵無法從照片中確認，明確指出
```

**預期效果**：減少模型「跳到結論」的傾向，強迫它先做仔細觀察

### 改進 4：Prompt 主體用英文、回覆要求中文

測試中發現模型用英文推理時可能更準確（訓練資料中英文植物學資料更豐富），
但使用者需要中文回覆。

```
You are a toxicology expert. Analyze the plant in this photo...
(英文主體)
...
Please respond in Traditional Chinese (繁體中文).
```

**狀態**：✅ 已間接驗證（T8 使用英文 Prompt + 中文回覆，效果良好）

### T9 關鍵發現

#### 發現 7：混淆鑑別的安全機制完美運作

兩張照片（一張姑婆芋、一張芋頭）都被判定為「無法確定」。
安全閾值規則（信心度 < 80% 或關鍵特徵無法確認 → 視為有毒）被嚴格遵守。
**在求生場景中，這是正確的行為。**

#### 發現 8：靜態照片的物理限制 — 揭示「互動式鑑別」的需求

> **核心問題：水珠測試是區分芋頭和姑婆芋的最關鍵指標，**
> **但這個測試需要使用者「在葉面滴水並觀察」，靜態照片永遠無法提供此資訊。**

這不是模型的問題，而是任務本身的物理限制。
這反而引出了一個非常重要的設計方向：

**系統不應該只被動接收照片，還應該主動引導使用者做互動式測試。**

設計方案：

```
使用者拍照 → 模型初步判斷 → 發現是混淆物種對
→ 系統主動引導：
  「⚠️ 此植物可能是芋頭（可食）或姑婆芋（有毒），
    僅靠照片無法區分。請執行以下測試：
    
    🔬 水珠測試：在葉面滴一滴水，觀察：
      → 水珠呈圓珠狀滾動 → 較可能是芋頭
      → 水珠攤平不成珠 → 較可能是姑婆芋
    
    📷 補拍照片：
      → 請拍攝葉柄與葉片的接合處特寫
      → 請拍攝葉面特寫（觀察是否光滑如蠟）
    
    拍攝後上傳，我將結合新資訊重新判斷。
    在完成測試前，請勿食用此植物。」
```

**這個設計讓系統從「被動辨識工具」進化為「主動求生教練」，是極大的加分項。**

#### 發現 9：模型對部分特徵的觀察準確度有限

- T9-A（姑婆芋）：模型認為葉面「略帶紋理/微絨毛」→ 實際姑婆芋是光滑的
- T9-B（芋頭）：模型認為葉柄「接在邊緣」→ 芋頭應該是盾狀著生

模型對精細質感和結構的判斷不夠準確。
但因為安全閾值的保護，這些誤判不會導致危險結論。

**開發指引**：不能只靠模型的視覺判斷來做最終決定，
必須搭配互動式測試（水珠測試、補拍照片）作為二次驗證。

## T10 測試結果（JSON 結構化輸出）

### 測試條件

- **照片**：姑婆芋（同 T8/T9）
- **Prompt 改進**：要求純 JSON 輸出，定義完整 schema
- **差異**：此 Prompt **未包含** T8 的詳細毒性植物特徵清單和 CoT 步驟

### 結果

**JSON 格式面 ✅**

- JSON 解析一次成功，無需清理
- 結構完全符合 schema 定義
- 所有 enum 值合規（`"high"`, `"RED"`, `"DO_NOT_TOUCH"`）
- JSON 前後沒有多餘文字
- 繁體中文正確使用

**辨識準確度 ❌**

- 模型輸出：「枯婆草 / Solanum spp.」（茄科）
- 實際答案：姑婆芋 / Alocasia odora（天南星科）
- **完全錯誤，而且信心度給了 80%**
- `confusion_pairs` 非常籠統：「其他常見的闊葉植物」
- `requires_interactive_test` 為 false（應為 true）

### 發現 10：JSON 格式與辨識準確度的衝突

> **當 Prompt 只要求 JSON 輸出而省略了詳細特徵清單和 CoT 步驟時，**
> **模型的辨識能力大幅退化——從 T8 的 85% 正確辨識，退化到完全辨錯。**

**原因分析**：

1. 強制 JSON 輸出讓模型跳過了「先觀察再推理」的思考過程
2. 沒有提供毒性植物特徵清單作為參考，模型只能從訓練資料中猜測
3. JSON 格式的約束讓模型更專注於「填格子」而非「做分析」

**設計指引**：

- **方案 A（推薦）**：兩階段輸出——先讓模型用 CoT 做完整分析，再要求整理成 JSON
- **方案 B**：在 JSON schema 中加入 `"reasoning"` 欄位，強迫模型在 JSON 內部先寫推理
- **方案 C**：保留 T8 的完整 Prompt（含特徵清單 + CoT），只在最後加上「將以上分析整理為 JSON」

**結論**：JSON 格式技術上可行，但 **不能犧牲 CoT 和特徵清單**。
正式版需要採用方案 A 或 C 來兼顧兩者。

### T10-v2：CoT + 特徵清單 + JSON（方案 C 驗證）

**改進**：保留 T8 完整 Prompt（CoT 四步驟 + 詳細特徵清單），
在最後加上 `<JSON_START>` / `<JSON_END>` 標記要求。

**結果**：

- ✅ JSON 解析成功（用 `<JSON_START>` 標記精確擷取）
- ✅ 正確辨識為姑婆芋 (Alocasia odora)
- ✅ 信心度 **90%**（比 T8 的 85% 還高！）
- ✅ CoT 步驟完整執行（Step 1-4 全部展開）
- ✅ `reasoning_summary` 欄位品質良好
- ⚠️ `confusion_pairs` 列出咬人狗而非芋頭（不夠精準）
- ⚠️ `requires_interactive_test` 仍為 false

### 發現 11：方案 C 完美解決 JSON vs CoT 的衝突

> **「先分析後整理」的策略成功了。**
> CoT 讓模型先做完深度思考，再將結論壓縮到 JSON 中。
> 信心度不但沒有下降，反而從 85% 提升到 90%。
> `<JSON_START>` / `<JSON_END>` 標記讓程式端的解析簡單可靠。

**開發指引**：正式版採用方案 C ——

## T14 測試結果（蛇類照片辨識）

### 測試條件

- **照片**：赤尾青竹絲（udn.com.tw 圖片）
- **Prompt**：爬蟲學專家角色 + 五種台灣常見毒蛇詳細特徵 + CoT + 急救建議

### 結果


| 物種        | 相似度       | 備註                  |
| --------- | --------- | ------------------- |
| **赤尾青竹絲** | **65%** ✅ | 正確鎖定目標，剛好觸發 ≥60% 警告 |
| 百步蛇       | 30%       | 正確排除                |
| 龜殼花       | 20%       | 正確排除                |
| 雨傘節       | 10%       | 正確排除（無黑白橫帶）         |
| 眼鏡蛇       | 5%        | 正確排除                |


### 關鍵觀察

1. **正確辨識核心特徵**：鮮綠色、中小型、細長體型、纏繞樹枝
2. **誠實標記無法確認的特徵**：
  - 尾巴末端是否為磚紅色（關鍵特徵）→ 照片遮擋無法確認
  - 瞳孔形狀（縱裂 vs 圓形）→ 解析度不足
  - 頰窩 → 無法確認
3. **警告觸發正確**：65% ≥ 60%，發出完整警告
4. **急救建議完整且正確**：立即就醫、保持冷靜、控制出血、勿自行處理

### 發現 13：蛇類辨識策略同樣有效

> **和植物辨識一樣的模式：模型能正確辨識大類特徵（綠色蛇 → 青竹絲方向），**
> **但對精細特徵（尾部顏色、瞳孔形狀）需要更高品質的照片。**
> **保守的 65% 信心度是合理的——照片確實無法看清關鍵特徵。**

**開發指引**：

- 蛇類的 Prompt 策略（詳細特徵 + CoT）和植物一致，可共用架構
- 蛇類辨識同樣需要「引導拍照」：建議拍攝頭部特寫、尾部、全身
- 急救建議模板可以預設在知識庫中，不需每次由模型生成
